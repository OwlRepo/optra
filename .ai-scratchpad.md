## Fixes — TDD order, failing test first

### F1 (P1) — Frontend: show source transcript + surface fieldConfidence
`apps/web/app/workspaces/[id]/tickets/page.tsx` (+ `page.spec.ts`)
The approved design (plan-design-review IA decision + CEO expansion #4) requires the reviewer to verify extracted fields against the source, and to see extraction confidence. Both are missing though the data exists (`TicketDetail.transcript`, `TicketDetail.fieldConfidence`).
- Render the selected ticket's `transcript` read-only **beside** the editor fields (side-by-side on desktop/tablet, stacked/collapsible on mobile). It is currently fetched but never displayed.
- Render a confidence summary badge ("N/7 high confidence", where high = `fieldConfidence[field] >= 0.7`) at the top of the draft.
- Give `hypothesizedRootCause` an amber "best guess — verify" affordance when its confidence is low (`< 0.7`), using existing `@repo/ui` tokens (no new colors).
- Tests (Vitest/jsdom, write first): assert the transcript text renders read-only when a ticket is selected; assert the confidence badge shows the correct N/7; assert the root-cause low-confidence affordance appears when `fieldConfidence.hypothesizedRootCause < 0.7` and not when `>= 0.7`.

### F2 (P1) — Processor: guard the completion write against clobbering a reviewed row
`apps/api/src/tickets/ticket-extraction.processor.ts:34-42` (+ `ticket-extraction.processor.spec.ts`)
The done-update spreads `...extracted` with no status guard. There is no re-queue path today, but add cheap insurance so a future retry can never overwrite a human's edits.
- Add `WHERE id = :ticketId AND status IN ('pending','processing')` to the success UPDATE (Drizzle `and(eq(id), inArray(status, ['pending','processing']))`). If 0 rows updated, log and no-op (row was already terminal/edited).
- Test (write first): a ticket already `done` with human-edited fields → re-running the processor does NOT overwrite the edited fields.

### F3 (P1) — Service: pending stale threshold must exceed job timeout
`apps/api/src/tickets/tickets.service.ts:15,17` (+ `tickets.service.spec.ts`)
`PENDING_TICKET_STALE_MS` is 2 min but `TICKET_JOB_TIMEOUT_MS` is 5 min — a valid job waiting in a backlog can be marked `failed` at 2 min.
- Raise `PENDING_TICKET_STALE_MS` to comfortably exceed the job timeout (e.g. `10 * 60_000`).
- Test (write first): a `pending` row whose `enqueuedAt` is 3 min old with no Bull job is NOT reconciled to `failed` (still within grace); one older than the new threshold with no job IS.

### F4 (P2) — Schema+Service: make dedup atomic
`packages/db/src/schema/tickets.ts:48`, new migration `0008_*`, `apps/api/src/tickets/tickets.service.ts:37-64` (+ service spec)
`transcript_hash` is a plain index; check-then-insert is a TOCTOU race → concurrent identical POSTs create duplicate drafts + double LLM spend.
- Change the index to a **unique** index on `(workspace_id, transcript_hash)`; generate migration `0008` via `bun run --cwd packages/db db:generate`.
- Keep the existing pre-check as a fast path, but wrap the insert in a try/catch: on a unique-violation, re-fetch and return the existing ticket as the 200-reuse response (same shape as the pre-check hit).
- Test (write first): two `create` calls with the same transcript in the same workspace → exactly one row; second returns statusCode 200 reuse. (Simulate the violation by inserting the row between the pre-check and insert, or by asserting the catch path returns the existing ticket.)

### F5 (P2) — DTO: cap the unbounded PATCH text fields
`apps/api/src/tickets/dto/update-ticket.dto.ts` (+ e2e or service spec)
`issueSummary`, `reproSteps`, `hypothesizedRootCause`, `nextAction`, `feedbackNote` have no `@MaxLength` (only `title` 500 and `productArea` 120 do). A member can PATCH multi-MB strings.
- Add `@MaxLength(...)` to each (pick sane caps, e.g. 4000 for summary/repro/root-cause/next-action, 2000 for feedbackNote — match the chat DTO's 4000 convention where reasonable).
- Test (write first): a PATCH exceeding a field's cap → 400.

### F6 (P2) — Service: stop returning internal queue fields from getOne
`apps/api/src/tickets/tickets.service.ts:91-103` (+ service spec)
`getOne` returns `select()` (every column). The UI needs `transcript` and `lastError` (both are shown by design), but does NOT need the internal queue plumbing.
- Project the `getOne` result to omit `queueJobId`, `transcriptHash`, `enqueuedAt`, `processingStartedAt`. Keep everything the editor renders (transcript, all draft fields, fieldConfidence, status, lastError, feedback/audit fields).
- Test (write first): the `getOne` response object does not contain `queueJobId`/`transcriptHash`/`enqueuedAt`/`processingStartedAt`, and still contains `transcript` + `fieldConfidence`.
