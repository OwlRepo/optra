# Fix: chat cache never invalidates when ticket knowledge changes, and semantic cache has no expiry at all

Status: IMPLEMENTATION_READY

Deep implementation approved: Yes — user reviewed and approved the plan.

## 1. Feature Selected
Two related fixes, bundled because the second was discovered while root-causing the first and shares the same root cause class (chat cache staleness):

**Fix A — ticket chunk changes never bump the cache version.** `syncTicketChunk()` inserts/deletes rows in `chunks` when a ticket's embed-qualifying state changes, but nothing bumps `chat_cache`'s per-workspace version when that happens. Document-side chunk mutations (`syncChunks()`, called from `ingest.processor.ts`) already call `cache.bumpVersion(workspaceId)` on every success — ticket-side mutations never call it, on either path that can trigger them: the live review-save flow (`tickets.service.ts#update()`) and the one-off `backfillTicketEmbeddings()` script.

**Fix B — the semantic cache (`chat_cache` table) never expires on its own.** The exact-match Redis cache has a real TTL (`CHAT_CACHE_TTL_SECONDS`, default 3600s). The semantic cache has none — a row written today is still eligible to be served as a "hit" a year from now, as long as `workspace_id` and `version` still match. Version bumps (including the fix in Fix A) are the *only* invalidation path today; if any future code path ever mutates retrievable content without remembering to bump the version — exactly the class of bug Fix A closes for tickets — the semantic cache has zero fallback protection. This is a deliberate design gap being closed now, not deferred, per explicit instruction to dig into it and include the fix in this same plan.

Reproduced Fix A live and root-caused with exact timestamps: a ticket was reviewed+embedded at `06:57:10`; 5 test questions asked at `06:55:53`-`06:56:44` (before the ticket existed) correctly got cached "I don't have enough information" answers; a 6th question asked at `07:00:18` (after the ticket existed) got a correct grounded answer, proving retrieval itself works. But the 5 earlier cached misses are now permanently stuck without Fix A — any matching or semantically-close question keeps hitting the stale cached answer forever.

## 3. Current Data / Control Flow (traced from source)
- `tickets.service.ts#update()` calls `syncTicketChunk(updated)` unconditionally after every successful review-save, wrapped in `.catch((err) => { this.logger.warn(...) })` that currently discards the outcome.
- `backfillTicketEmbeddings()` loops over every ticket in the DB, calls `syncTicketChunk()` per row, tallies global counts, has no per-workspace breakdown and no cache-layer awareness (by design — `packages/ai` has zero Redis/cache coupling, a boundary worth preserving).
- `scripts/backfill-ticket-embeddings.ts` is a 12-line CLI wrapper running outside NestJS DI — no access to `CacheService`.
- `cache.service.ts#getSemantic()` is called from `chat.service.ts` on every chat request as the second cache-lookup tier (after exact-match misses, before falling through to real generation). Any semantic hit, regardless of age, short-circuits generation and returns the stored answer/sources verbatim.
- `cache.service.ts#saveSemantic()` is called from `chat.service.ts` after a genuine cache-miss generation completes, to store the new answer for future semantic lookups. This is the natural, already-existing point where opportunistic cleanup of that workspace's own expired rows can piggyback with zero new infrastructure.

## 4. Feature Gap Analysis
1. `tickets.service.ts#update()` never bumps the cache version, even when `syncTicketChunk()` genuinely changes `chunks` state. (Fix A)
2. `backfillTicketEmbeddings()` provides no way for its caller to know which workspaces had a real chunk change, so the script can't bump the right versions even if it wanted to. (Fix A)
3. `scripts/backfill-ticket-embeddings.ts` has no cache-invalidation capability at all today. (Fix A)
4. `chat_cache` rows never expire by age — `getSemantic()` will serve an arbitrarily old row as a valid hit as long as version matches, with no time-based fallback if a version bump is ever missed for any reason, now or in the future. (Fix B)
5. `saveSemantic()` never removes old rows — the table grows without bound over time; not itself a correctness bug, but the natural side effect of fixing gap 4 with an opportunistic-cleanup design is that this also gets addressed as a side effect, not a separately engineered feature.

### Fix A — `packages/ai/src/vectorstore/index.ts`
Extend `backfillTicketEmbeddings()`'s return type and body to also report which workspaces actually changed, without adding any cache/Redis dependency to this file:
```ts
export async function backfillTicketEmbeddings(): Promise<{
  processed: number
  embedded: number
  deleted: number
  skipped: number
  unchanged: number
  changedWorkspaceIds: string[]
}> {
  const allTickets = await db.select().from(tickets)
  const result = { processed: 0, embedded: 0, deleted: 0, skipped: 0, unchanged: 0 }
  const changedWorkspaceIds = new Set<string>()

  for (const ticket of allTickets) {
    const outcome = await syncTicketChunk(ticket)
    result.processed += 1
    result[outcome] += 1
    if (outcome === 'embedded' || outcome === 'deleted') {
      changedWorkspaceIds.add(ticket.workspaceId)
    }
  }

  return { ...result, changedWorkspaceIds: [...changedWorkspaceIds] }
}
```

```ts
import Redis from 'ioredis'
import { backfillTicketEmbeddings } from '@repo/ai'

async function bumpCacheVersions(workspaceIds: string[]): Promise<void> {
  if (workspaceIds.length === 0) return

  const redis = new Redis({
    host: process.env.REDIS_HOST || 'localhost',
    port: Number.parseInt(process.env.REDIS_PORT || '6379', 10),
  })

  try {
    for (const workspaceId of workspaceIds) {
      await redis.incr(`chat:ver:${workspaceId}`)
    }
  } finally {
    await redis.quit()
  }
}

backfillTicketEmbeddings()
  .then(async (result) => {
    console.log(result)
    await bumpCacheVersions(result.changedWorkspaceIds)
    process.exit(0)
  })
  .catch((error) => {
    console.error(error)
    process.exit(1)
  })
```

Update `saveSemantic()` to opportunistically delete this workspace's own expired rows right after a successful insert, with the cleanup's own independent try/catch so a cleanup failure is logged distinctly and never masks whether the actual cache write succeeded:
```ts
  async saveSemantic(
    workspaceId: string,
    version: number,
    message: string,
    embedding: number[],
    answer: string,
    sources: ChatMessageSource[],
  ): Promise<void> {
    try {
      await db.insert(chatCache).values({
        workspaceId,
        version,
        question: message,
        questionEmbedding: embedding,
        answer,
        sources,
      })
    } catch (error) {
      this.logger.warn(`Failed semantic chat cache write for workspace ${workspaceId}: ${this.message(error)}`)
      return
    }

    try {
      await this.deleteExpiredSemanticCache(workspaceId)
    } catch (error) {
      this.logger.warn(`Failed expired semantic chat cache cleanup for workspace ${workspaceId}: ${this.message(error)}`)
    }
  }

  private async deleteExpiredSemanticCache(workspaceId: string): Promise<void> {
    const ttlHours = this.semanticCacheTtlHours()
    await db.execute(sql`
      delete from chat_cache
      where workspace_id = ${workspaceId}
        and created_at <= now() - make_interval(hours => ${ttlHours})
    `)
  }
```
This is opportunistic, not a guaranteed sweep: a workspace's expired rows only get physically deleted the next time that same workspace writes a new semantic cache entry. A workspace that goes permanently idle keeps its expired (but already-unreadable-by-`getSemantic()`'s-own-age-filter) rows sitting in the table indefinitely. That's a storage-hygiene tradeoff, not a correctness one — `getSemantic()`'s own age filter is what actually closes the staleness bug; the delete-on-write is a bonus that keeps the common case clean without needing a scheduler. Stated honestly per section 2's note, not hidden as a complete solution.

## 12. Rollback / Risk Mitigation Plan
- Purely additive for both fixes, no schema, no data migration. Rollback = revert the commit.
- Fix A risk: over-bumping is the safe failure direction if the conditional check is ever wrong (a few extra cache misses, never wrong answers). Under-bumping is the bug being fixed — the conditional check is deliberately narrow (`'embedded' | 'deleted'` only).
- Fix B risk: TTL duration is a real tuning knob, not a free parameter — too short (e.g. under an hour) meaningfully hurts semantic-cache hit rate and therefore cost/latency for legitimately-still-valid answers; too long re-widens the staleness window this fix exists to bound. `24` hours is a reasonable default (long enough to catch real repeat/paraphrased traffic across a work day, short enough to bound worst-case staleness to about a day even if a future bump-path gets missed) but should be revisited with real usage data, not treated as permanently correct. Configurable via env var specifically so it can be tuned without a redeploy.
- Fix B is explicitly NOT a full solution to unbounded table growth (see section 7's honesty note on opportunistic-vs-scheduled cleanup) — a genuinely idle workspace's expired rows persist physically until that workspace's next semantic-cache write. If this ever becomes an actual storage problem (not currently evidenced), the real fix at that point is introducing `@nestjs/schedule` or an external cron, which is new infrastructure this plan deliberately does not introduce without that being its own explicit decision.
- No deployment ordering risk for either fix — self-contained within `apps/api` + `packages/ai`, no cross-service coordination.
