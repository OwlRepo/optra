# Fix: fallback/non-answers get cached as if they were real answers

Status: IMPLEMENTATION_READY

Deep implementation approved: Yes — user reviewed and approved the plan.

## 1. Feature Selected
Root-caused as the third layer of a "every chat question returns the fallback message forever" bug this session (layers 1-2 — wrong execution mode, miscalibrated `RETRIEVAL_SCORE_THRESHOLD` — already fixed operationally, verified live). `apps/api/src/chat/chat.service.ts`'s cache-miss `onComplete` callback (`chat.service.ts:78-93`) calls `cache.setExact(...)` and `cache.saveSemantic(...)` unconditionally on whatever text came back from generation, with no check that it was a real, grounded answer. Confirmed directly in Redis during the live investigation: a cache key under the *current* version held `{"answer":"I don't have enough information to answer that. Consider escalating to a human.","sources":[]}` as if it were a cached success. Once a phrasing gets poisoned this way, it stays stuck returning that answer regardless of whether live retrieval would now succeed for it — independent of and compounding whatever originally caused the bad answer.

## 2. Existing System Discovery
- `packages/ai/src/chains/index.ts` — `AnswerResult` interface (`index.ts:27-30`) currently `{ sources: ChatSource[]; stream: AsyncGenerator<string> }`. Two return points inside `answerQuestion()`: the `chunks.length === 0` early return (`index.ts:44-51`, hardcoded fallback text, no chunks retrieved at all) and the normal end-of-function return after a real `generate` call (`index.ts:137-153`).
- `packages/ai/src/chains/graph.ts` — **important finding, changes the exact field name used**: `GraphState` already has a field called `grounded` (`graph.ts:41`, `Annotation<boolean | undefined>`), set by `gradeAnswerNode()` (`graph.ts:230-232`) as part of the *optional* self-grade quality-check feature (`SELF_GRADE_ENABLED`, off by default) — a completely different concept (LLM's own quality judgment on an already-real answer) from what this fix needs (was *any* real answer attempted at all, vs. the deterministic fallback path). Reusing the name `grounded` for the new field would collide in meaning within the same file even though they're different state slots — picking a distinct name (`isFallback`) to avoid that confusion for whoever reads this later.
- `graph.ts`'s `fallbackNode()` (`graph.ts:250-255`) is the terminal node reached via `routeAfterRetrieve()` (`graph.ts:177-189`) when retrieval score never clears `retrievalThreshold()` after exhausting `maxQueryRewrites()` attempts — returns `{ answerText: FALLBACK_MESSAGE, sources: [] }`. This is the exact path that was firing on every question before today's threshold fix, confirmed live.
- `answerQuestionWithGraph()` (`graph.ts:273-294`) is the single point where graph state collapses back into the shared `AnswerResult` shape returned to `chains/index.ts`'s caller — the only place that needs to read the graph's fallback signal and translate it outward.
- `apps/api/src/chat/chat.service.ts` (read in full) — confirmed exact current shape of the cache-miss branch (`chat.service.ts:69-94`); `const { sources, stream } = await answerQuestion(message, workspaceId)` destructures only `sources`/`stream` today, `AnswerResult`'s new field just needs to also be destructured here.
- **Deliberately out of scope, stated honestly, not silently dropped**: the model can also produce the *exact same* "I don't have enough information to answer that." text on its own, in-band, via the normal `generate` path — when real chunks were retrieved and fed as context, but the LLM judges (per its own system prompt instruction) that the context doesn't actually answer the question. This happened earlier this session on a real, correctly-cited retrieval. That case is NOT covered by this fix — structurally, it went through the real `generate` branch, not a hardcoded fallback path, so `isFallback` would correctly read `false` for it. Catching it would require either fragile string-matching against the fallback phrase (already rejected as an approach for the deterministic case, for the same reason) or a second self-grading LLM call (real cost/latency, arguably a different feature). Not solving it here — flagging it as a known, narrower, softer edge case for a future decision, not pretending this fix is exhaustive.

## 3. Current Data / Control Flow (traced from source)
`chat.service.ts#answer()`'s cache-miss branch (`chat.service.ts:69-94`) is the *only* place either `cache.setExact()` or `cache.saveSemantic()` gets called from generated (non-cache-hit) text — both calls sit inside the same `onComplete` closure, unconditionally, right after `persistAssistant()`. Both need the same guard, in the same place, since they're the same decision applied to two different cache layers (exact-Redis, semantic-Postgres).

## 4. Feature Gap Analysis
1. `AnswerResult` carries no signal distinguishing "real generated answer" from "deterministic fallback, no generation attempted/trusted."
2. `chat.service.ts` has no way to conditionally skip caching even if it wanted to, since that signal doesn't exist yet.
3. Both `chains/index.ts` and `graph.ts` need to populate the same new field consistently, since either can be the active path depending on `LANGGRAPH_ENABLED`.

## 5. API Contract Plan
No API contract changes required. No new endpoint, no request/response shape change to any client-facing route. `isFallback` is internal to the answer-generation-to-cache-write path inside `apps/api`/`packages/ai` — never persisted to `chat_messages.sources`/`chat_cache.sources`, never sent to the frontend.

Update `answerQuestionWithGraph()`'s initial state object (current `graph.ts:278-286`) to seed the new field, and its return (current `graph.ts:288-293`) to surface it:
```ts
export async function answerQuestionWithGraph(
  question: string,
  workspaceId: string,
  limit = 5,
): Promise<AnswerResult> {
  const result = await graph.invoke({
    question,
    workspaceId,
    limit,
    rewrites: 0,
    chunks: [],
    sources: [],
    regenerated: false,
    isFallback: false,
  })

  return {
    sources: result.sources ?? [],
    isFallback: result.isFallback ?? false,
    stream: (async function* () {
      yield result.answerText ?? FALLBACK_MESSAGE
    })(),
  }
}
```
Every other node (`generateNode`, `regenerateNode`, `gradeAnswerNode`, `rewriteNode`) never sets `isFallback`, so it stays at its seeded `false` through the whole graph invocation unless `fallbackNode` specifically runs — correct, since those are all real-generation paths.

### `apps/api/src/chat/chat.service.ts`
Update the cache-miss branch (current `chat.service.ts:69-94`) to destructure the new field and gate both cache writes on it:
```ts
    const version = await this.cache.getVersion(workspaceId)
    await this.usage.assertWithinBudget(workspaceId)
    const { sources, stream, isFallback } = await answerQuestion(message, workspaceId)

    return {
      sessionId: session.id,
      sources,
      stream,
      cacheStatus: 'miss' as CacheStatus,
      onComplete: async (fullText: string) => {
        await this.usage.addUsage(
          workspaceId,
          countTokens(message) + countTokens(fullText),
        )
        await this.persistAssistant(session.id, fullText, sources)

        if (!isFallback) {
          await this.cache.setExact(workspaceId, message, fullText, sources)
          await this.cache.saveSemantic(
            workspaceId,
            version,
            message,
            embedding,
            fullText,
            sources,
          )
        }
      },
    }
```
`persistAssistant()` still runs unconditionally — a fallback answer still belongs in the user's own chat history (they asked, they got a real response, just not a grounded one), it just shouldn't be reused as a cached "answer" for other/future askers of a similar question. Only the two cache-write calls move inside the new guard.

## 12. Rollback / Risk Mitigation Plan
- Purely additive (one new field, one new conditional), no schema, no data migration. Rollback = revert the commit.
- Risk: if `isFallback` is ever left unset/`undefined` on some future new return path added later, it needs to default to `false` (real answer) or `true` (fallback) deliberately depending on that path's nature — not left ambiguous. Both current implementations seed it explicitly at every return point, no implicit fallthrough.
- No deployment ordering risk — self-contained within `apps/api` + `packages/ai`.
