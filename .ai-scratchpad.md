# Feature: Embed reviewed tickets into pgvector RAG

Status: IMPLEMENTATION_READY

Deep implementation approved: Yes — user approved chunking/backfill design decisions, then explicitly approved this final plan and requested Codex handoff.

## 1. Feature Selected
Embed `tickets` rows meeting `status='done' AND reviewedBy IS NOT NULL AND usefulness='useful'` into the same vector store Chat/RAG already reads, so past resolved tickets can be cited in Chat answers alongside KB documents. Covers a one-off backfill of existing qualifying tickets plus ongoing embedding as tickets get reviewed going forward. User-approved design decisions: (1) one chunk per ticket, not multi-chunk; (2) backfill via a one-off script, not an admin endpoint.

## 2. Existing System Discovery
No prior art — zero refs to embed/chunk/vector anywhere in tickets code (verified via grep across `apps/api/src/tickets/*`, `packages/ai/src/chains/ticket-extraction.ts`).

Reusable patterns (verified from source):
- `syncChunks()` + `embedChunks()` in `packages/ai/src/vectorstore/index.ts` — content-hash diff, idempotent. Currently document-only (see Gap 1).
- `similaritySearch(query, workspaceId, limit)` — single shared retrieval fn, already workspace-scoped, no schema change needed on the read side beyond what Gap 4 requires.
- Precedent for a standalone backfill/eval script living outside app runtime: `scripts/eval/*.py`. This feature follows the same shape but in TS (`scripts/backfill-ticket-embeddings.ts`), since it needs `@repo/db` + `@repo/ai`.

## 3. Current Data / Control Flow (traced from source)
- `chunks.documentId` is `uuid(...).references(documents.id).notNull()` — hard FK, chunks table is document-only by construction today.
- Ticket `status` only ever becomes `'done'` from `ticket-extraction.processor.ts#handleExtraction` (line 47). `reviewedBy` / `usefulness` are only ever set from `tickets.service.ts#update()` (PATCH review save, `tickets.service.ts:188-209`). So the qualifying transition can only happen inside `update()`, and only after extraction already completed — confirms the hook location.
- 3 call sites duplicate the identical citation-hydration guard `if (!documentId) continue`: `packages/ai/src/chains/index.ts:63-66`, `packages/ai/src/chains/graph.ts:57-60`, `apps/api/src/search/search.service.ts#searchDocuments:55-58`. All three would silently and safely ignore ticket-authored chunks IF we key them on `metadata.ticketId` instead of `metadata.documentId` — confirmed this means **zero Search-domain risk** (no code change needed in `search.service.ts`).
- `buildContext()` in `chains/index.ts:18-21` does **not** filter by metadata — it feeds every retrieved chunk's content to the LLM. This means the moment ticket chunks exist in `chunks`, their content enters answers immediately, whether or not citation hydration has been updated yet. Deployment ordering must land the citation-hydration fix (Gap 4) in the **same deploy** as the chunk-insert capability, not after.

## 4. Feature Gap Analysis
1. **Schema**: `chunks.documentId` NOT NULL FK blocks non-document rows. Needs nullable `documentId` + new nullable `ticketId` FK + CHECK constraint (exactly one of the two set).
2. **Embed/un-embed hook**: nothing watches ticket review-save for qualifying-state transitions. New hook in `tickets.service.ts#update()`.
3. **Backfill**: nothing embeds tickets that already qualify today. New one-off script.
4. **Citation contract**: `ChatSource` (`chains/index.ts:23-29`), `ChatMessageSource` (`packages/db/src/schema/chatSessions.ts:5-11`), and FE `ChatSource` (`apps/web/app/workspaces/[id]/chat/page.tsx:35-40`) are three duplicated definitions of the same shape, all document-only. All three need a ticket variant. Backward compat: `chat_messages.sources` / `chat_cache.sources` rows written before this ships have no `sourceType` field at all. Only the FE (`page.tsx`) is the actual runtime read boundary for that old data (server-side, `sources` is always treated as an opaque array, never destructured field-by-field — confirmed in section 7), so only the FE type needs to model the "field may be missing" case; the two server-side types can stay strict discriminated unions. See sections 7-8 for the exact resolution.

## 5. API Contract Plan
No new HTTP endpoint, no request/response wire-shape change to any listed route. Internal-only contract change: the `sources` array's *item* shape gains a discriminant. Update `docs/ai/contracts/api-contracts.md` Notes column for the "Chat / RAG | Stream Answer" and "Chat / RAG | List Messages" rows to say citations may now be ticket-sourced (`sourceType: 'ticket'`), not document-only.

## 6. Database & Schema Changes

### Migration Danger Gate (final answers)
- Migration required? Yes.
- Backfill required? Yes, separate step (script), not part of the schema migration itself.
- Default/nullability? `chunks.document_id` becomes nullable. Verified 100% of existing rows have `document_id NOT NULL` and no `ticket_id` column exists yet, so the new CHECK constraint (`(document_id IS NOT NULL) <> (ticket_id IS NOT NULL)`) is satisfied trivially by every existing row — no data backfill needed to make the constraint valid.
- Index/constraint impact: new nullable FK `ticket_id -> tickets.id ON DELETE CASCADE`; new **unique** index on `ticket_id` (enforces the "one chunk per ticket" design decision at the DB level, nullable-safe — Postgres unique indexes allow multiple NULLs); new CHECK constraint.
- Existing data impact: none — no ticket-linked chunk rows exist anywhere yet.
- Rollback possible? Yes pre-launch: drop the CHECK, drop `ticket_id` column, restore `document_id` NOT NULL — safe only because no ticket-authored rows will exist to violate the NOT NULL revert, i.e. rollback must happen before or instead of running the backfill script.
- Deployment ordering risk: migration → app code (hook + citation fix) → backfill script, in that order. Running the backfill before the migration lands, or before the citation-hydration fix ships, is the one sequencing mistake to avoid (see Gap 3 note in section 3).

**`syncTicketChunk(ticket: Ticket): Promise<'embedded' | 'deleted' | 'unchanged' | 'skipped'>`** (exported) — the ONE function both the review-save hook and the backfill script call. **Embed before delete** — if the embedding API call fails, the previous (stale but present) chunk must survive untouched rather than being deleted first and orphaning the ticket un-cited:
```ts
export async function syncTicketChunk(
  ticket: Ticket,
): Promise<'embedded' | 'deleted' | 'unchanged' | 'skipped'> {
  const qualifies =
    ticket.status === 'done' && ticket.reviewedBy !== null && ticket.usefulness === 'useful'

  const [existing] = await db
    .select({ id: chunksTable.id, contentHash: chunksTable.contentHash })
    .from(chunksTable)
    .where(eq(chunksTable.ticketId, ticket.id))
    .limit(1)

  if (!qualifies) {
    if (!existing) return 'skipped'
    await db.delete(chunksTable).where(eq(chunksTable.ticketId, ticket.id))
    return 'deleted'
  }

  const content = buildTicketChunkContent(ticket)
  const contentHash = createHash('sha256').update(content).digest('hex')

  if (existing && existing.contentHash === contentHash) {
    return 'unchanged'
  }

  const embedding = await embedQuery(content)

  await db.delete(chunksTable).where(eq(chunksTable.ticketId, ticket.id))
  await db.insert(chunksTable).values({
    ticketId: ticket.id,
    documentId: null,
    workspaceId: ticket.workspaceId,
    content,
    contentHash,
    embedding,
    metadata: { ticketId: ticket.id, workspaceId: ticket.workspaceId, source: 'ticket' },
  })

  return 'embedded'
}
```

**`backfillTicketEmbeddings(): Promise<{processed: number; embedded: number; deleted: number; skipped: number; unchanged: number}>`** (exported) — `db.select().from(tickets)`, loop, call `syncTicketChunk` per row, tally return values, return summary. **No `workspaceId` parameter — intentionally queries across ALL workspaces in one pass**, matching this being a one-off cross-tenant maintenance script, not a per-request path (do not add workspace scoping here, unlike every other query in this codebase). No qualifying pre-filter in the query itself — reuses the same `syncTicketChunk` decision logic so there is exactly one place that defines "qualifies," not two.
```ts
export async function backfillTicketEmbeddings() {
  const allTickets = await db.select().from(tickets)
  const result = { processed: 0, embedded: 0, deleted: 0, skipped: 0, unchanged: 0 }

  for (const ticket of allTickets) {
    const outcome = await syncTicketChunk(ticket)
    result.processed += 1
    result[outcome] += 1
  }

  return result
}
```

## 9. External Integration / Background Job Steps
- Integration: one extra OpenAI embedding call (`text-embedding-3-small`, same model already used for documents) per qualifying review-save — negligible cost, same integration surface already in production for documents.
- Job queue usage: none — this runs synchronously (fire-and-forget) inside the existing `PATCH .../tickets/:ticketId` request path, not a new Bull queue. No new queue = no new queue-reliability surface (no job id, no retry/backoff, no reconciliation needed) — deliberately simpler than the ingest/scrape/ticket-extraction queues since the operation is cheap (one embedding call) and idempotent (safe to just let the next review-save retry it if it silently failed).
- Idempotency: `syncTicketChunk()` is idempotent by construction — same ticket state in → same outcome, content-hash guards against redundant embedding calls.
- Error handling: `.catch()` + logger warn at the call site in `tickets.service.ts#update()`, matching the existing `EventsService.record()` fire-and-forget pattern — a vectorstore failure never fails the review-save HTTP response.
- Retry logic: none automatic. A failed `syncTicketChunk()` call self-heals on the ticket's *next* review-save (any subsequent `update()` call re-evaluates from scratch), or can be manually re-run via the backfill script (safe to re-run any time — see Rollback section).

## 12. Rollback / Risk Mitigation Plan
- If the migration needs reverting before the backfill script has ever run: drop the CHECK constraint, drop `ticket_id` column + its unique index, restore `document_id NOT NULL` — safe, no data loss, no ticket-authored rows will exist yet.
- If reverting AFTER the backfill has run: ticket-authored `chunks` rows exist with `document_id IS NULL`; must delete `WHERE ticket_id IS NOT NULL` before restoring the `document_id NOT NULL` constraint, or that revert migration will fail outright on those rows.
- Data impact: purely additive (new `chunks` rows); nothing about existing document-chunk rows or existing chat history changes shape until read through the new discriminated-union type, which defaults old data safely.
- Deployment ordering risk (repeated from section 6/10 since it's the one real risk here): the citation-hydration fix (`chains/index.ts` + `graph.ts`) MUST ship in the same deploy as the schema+hook, not after — `buildContext()` has no filter, so a window where ticket chunks exist but citation code hasn't been updated yet means Chat could ground answers on ticket content while showing zero citation for it (silent, hard-to-notice regression in answer traceability, not a crash).
- If `syncTicketChunk` misbehaves in production (e.g. bad embeddings), the fix is: correct the function, then re-run `backfillTicketEmbeddings()` — it's idempotent and will re-derive every qualifying ticket's chunk from scratch based on current content hashes.
