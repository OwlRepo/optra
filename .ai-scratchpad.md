# Feature: Reserved ticket slot in chat retrieval (fix ticket citations getting drowned out by KB doc volume)

Status: IMPLEMENTATION_READY

Deep implementation approved: Yes — user reviewed and approved the plan.

## 1. Feature Selected
Chat retrieval (`answerQuestion()` / `answerQuestionWithGraph()`) currently pulls its top-K chunks from one shared pool across ALL chunk types (documents + tickets) via `similaritySearch()`. In a workspace with thousands of document chunks and only a handful of ticket chunks, a genuinely-correct ticket chunk can lose the ranking race purely on volume, even when it's the best answer — reproduced live this session: a workspace with 4,230 document chunks and 1 relevant ticket chunk (score 0.51, manually confirmed correct) failed to surface on a generic phrasing of the same question that a closely-worded phrasing later did surface correctly.

Fix: reserve a small number of top-K slots specifically for ticket chunks, gated by a minimum-relevance floor so irrelevant tickets never get force-fed into context. Not a UI filter (rejected — adds cognitive load, doesn't fix retrieval, discussed and dismissed with the user this session). Not a blanket "0.6 minimum score" cutoff (rejected — would have excluded the very citation that was manually confirmed correct at 0.51; there's no universal "good" cosine-similarity number, it's corpus- and model-specific).

## 4. Feature Gap Analysis
1. No source-type-aware retrieval exists — `similaritySearch()` is type-blind by design (correct for Search, wrong for Chat when corpus is imbalanced).
2. No mechanism reserves ticket representation in chat's top-K regardless of raw score competition.
3. No relevance floor exists to prevent an irrelevant ticket from being force-included once a reservation mechanism exists (needed so the fix doesn't trade "tickets never show" for "tickets show even when irrelevant").
4. Two existing test files mock `'../vectorstore'` narrowly enough that this change breaks them if not updated in the same commit (see section 3).

## 7. Backend Implementation Steps

### `packages/ai/src/vectorstore/index.ts`
Add two small tunable getters (same shape/placement convention as `graph.ts:159-165`) and one new exported function, placed after the existing `similaritySearch` function (do not modify `similaritySearch` itself):

```ts
function ticketSlotReserve(): number {
  return Number.parseInt(process.env.TICKET_SLOT_RESERVE ?? '1', 10)
}

function ticketSlotMinScore(): number {
  return Number.parseFloat(process.env.TICKET_SLOT_MIN_SCORE ?? '0.3')
}

export async function similaritySearchWithTicketSlot(
  query: string,
  workspaceId: string,
  limit = 5
): Promise<SearchResult[]> {
  const queryVector = await embedQuery(query)
  const vectorString = `[${queryVector.join(',')}]`
  const reserve = ticketSlotReserve()
  const minScore = ticketSlotMinScore()

  const documentRows = await db.execute<{
    id: string
    content: string
    metadata: Record<string, unknown>
    score: number
  }>(sql`
    SELECT
      id,
      content,
      metadata,
      1 - (embedding <=> ${vectorString}::vector) AS score
    FROM chunks
    WHERE workspace_id = ${workspaceId}::uuid AND document_id IS NOT NULL
    ORDER BY embedding <=> ${vectorString}::vector
    LIMIT ${limit}
  `)

  const ticketRows = await db.execute<{
    id: string
    content: string
    metadata: Record<string, unknown>
    score: number
  }>(sql`
    SELECT
      id,
      content,
      metadata,
      1 - (embedding <=> ${vectorString}::vector) AS score
    FROM chunks
    WHERE workspace_id = ${workspaceId}::uuid AND ticket_id IS NOT NULL
    ORDER BY embedding <=> ${vectorString}::vector
    LIMIT ${reserve}
  `)

  const documentResults: SearchResult[] = documentRows.rows.map(row => ({
    id: row.id,
    content: row.content,
    metadata: row.metadata ?? {},
    score: row.score,
  }))

  const qualifyingTicketResults: SearchResult[] = ticketRows.rows
    .filter(row => row.score >= minScore)
    .map(row => ({
      id: row.id,
      content: row.content,
      metadata: row.metadata ?? {},
      score: row.score,
    }))

  if (qualifyingTicketResults.length === 0) {
    return documentResults.slice(0, limit)
  }

  const documentBudget = Math.max(limit - qualifyingTicketResults.length, 0)
  const merged = [...documentResults.slice(0, documentBudget), ...qualifyingTicketResults]

  return merged.sort((a, b) => b.score - a.score)
}
```

Notes on this exact implementation, so nothing gets "improved" or reinterpreted:
- Reuses the existing exported `SearchResult` interface (`vectorstore/index.ts:7-12`) — no new type.
- Exactly 2 SQL queries per call, both reusing the single `embedQuery()` result — never re-embeds. A 3rd query was considered (for backfilling a dropped ticket slot with an extra document) and deliberately rejected: always fetching `limit` documents up front and only trimming the slice when a ticket qualifies is simpler, avoids a conditional 3rd round-trip, and the total result count is always exactly `min(limit, available rows)` either way.
- **Known, accepted behavior difference vs. the old unified-pool `similaritySearch()` call**: a ticket chunk scoring *below* `minScore` is now unconditionally excluded, even in the rare case it would have coincidentally ranked in the old naive top-`limit` pool (e.g. if it scored higher than a very weak document match). This is intentional — the floor exists specifically to gate ticket inclusion by a minimum confidence — but it is a real, honest tradeoff, not a strict improvement in every possible case. State this plainly if asked, don't claim it's a pure win.

```ts
async function seedDocumentChunk(workspaceId: string, kbId: string, embedding: number[]) {
  const [document] = await db
    .insert(documents)
    .values({
      workspaceId,
      knowledgeBaseId: kbId,
      title: 'Doc',
      storageKey: `doc-${crypto.randomUUID()}.txt`,
      status: 'done',
    })
    .returning()

  const [chunk] = await db
    .insert(chunks)
    .values({
      workspaceId,
      documentId: document.id,
      content: 'doc content',
      contentHash: crypto.randomUUID().replace(/-/g, ''),
      embedding,
    })
    .returning()

  return { document, chunk }
}

async function seedTicketChunk(workspaceId: string, ticketId: string, embedding: number[]) {
  const [chunk] = await db
    .insert(chunks)
    .values({
      workspaceId,
      ticketId,
      content: 'ticket content',
      contentHash: crypto.randomUUID().replace(/-/g, ''),
      embedding,
    })
    .returning()

  return chunk
}
```

Deterministic vector recipe — do not invent different vectors, use exactly these three so cosine scores are predictable:
- `HIGH = new Array(1536).fill(1)` — identical to the query vector below → cosine distance 0 → score ≈ 1.0.
- `MID = new Array(1536).fill(1).map((v, i) => (i < 768 ? 1 : -1))` — half-aligned/half-opposed → cosine distance 1 (orthogonal) → score ≈ 0.0.
- `LOW = new Array(1536).fill(-1)` — exact opposite of the query vector → cosine distance 2 → score ≈ -1.0.
- Query vector for all these tests: `new Array(1536).fill(1)` (same as `HIGH`), set via `embedQueryMock.mockResolvedValueOnce(...)` immediately before each `similaritySearchWithTicketSlot(...)` call.

## 12. Rollback / Risk Mitigation Plan
- Purely additive change (new function, two call-site swaps, two test-mock fixes) — no schema, no data migration, no new persisted state. Rollback = revert the commit, nothing to undo at the data layer.
- Risk: if `TICKET_SLOT_MIN_SCORE` is misconfigured too low in production (e.g. `0`), tickets would always take a slot regardless of relevance, degrading answer quality by displacing a genuinely-better document. Mitigated by shipping with the coded default (`0.3`), not an env var set in deployment config initially — only tune via the eval harness (`scripts/eval/evaluate.py`) with real before/after data, not by guessing a new number.
- Risk: `TICKET_SLOT_RESERVE` set too high (e.g. `5` with `limit=5`) would zero out the document budget entirely for any workspace with qualifying tickets. Same mitigation — ship with default `1`, don't hand-tune without eval evidence.
- No deployment ordering risk — self-contained within `packages/ai`, no cross-service coordination needed.
