## RCA / Confirmed Facts (reproduced)

**Bug 1 — API runtime module resolution**
- `apps/api/tsconfig.json` `compilerOptions.paths`: `"@repo/ai": ["../../packages/ai/src"]`, `"@repo/db": ["../../packages/db/src"]`, `"@repo/types": ["../../packages/types/src"]`.
- `nest build`/`nest start` use **SWC**, which honors those paths and emits e.g. `require('../../../../packages/db/src')` from `apps/api/dist/auth/auth.service.js`. That path is the TS **source** dir (no `index.js`) → `Error: Cannot find module '../../../../packages/db/src'` (`MODULE_NOT_FOUND`). Reproduced on BOTH `bun run --cwd apps/api dev` and `bun run --cwd apps/api start`.
- The packages are real workspace deps: `apps/api/package.json` lists `@repo/ai`/`@repo/db`/`@repo/types` as `workspace:*`; each package.json has `main: ./dist/index.js`; `packages/db/dist/index.js` EXISTS. So bare `@repo/db` resolves correctly via node_modules — the `paths` aliases are what break it.
- Tests pass because jest uses its own module resolution (independent of these build-time paths); the compiled server was never executed.

**Bug 2 — langgraph/core version mismatch**
- Installed: `@langchain/langgraph@0.2.74`, `@langchain/core@0.2.36`, `@langchain/openai@0.2.11`, `@langchain/community@0.2.33`.
- langgraph 0.2.70–0.2.74 import `isToolMessage` from `@langchain/core/messages`; core 0.2.x (latest is 0.2.36) does NOT export it → `SyntaxError: ... does not provide an export named 'isToolMessage'` in `packages/ai/src/chains/graph.spec.ts`.
- langgraph's declared peer (`@langchain/core >=0.2.36 <0.3.0 || >=0.3.40 <0.4.0`) technically allows 0.2.36, but the actual code needs a newer core — a known LangChain footgun. `^0.2.39` in `packages/ai/package.json` let `bun install` (run during the A1 work) bump it to 0.2.74.
- `chains/index.ts` imports the graph LAZILY (`await import('./graph')`) and is flag-gated (`LANGGRAPH_ENABLED=false`), so Bug 2 does NOT affect API boot — but it breaks `packages/ai` tests and would break the feature when enabled.
