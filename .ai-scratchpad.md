# Change password (auth)

Status: IMPLEMENTATION_READY

Deep implementation approved: Yes

Task Size: Deep (auth domain, non-downgradable per repo default — mutates live credentials)

Contract Areas:
- API: 1 new Auth endpoint (`POST /auth/change-password`) — see docs/ai/contracts/api-contracts.md updates below
- Database: no schema changes — mutates existing `users.passwordHash` and `refresh_tokens.revokedAt` columns only
- Permissions: any authenticated user, self only (`JwtAuthGuard`, no additional role check — you can only change your own password)
- External integrations: No contract impact.
- Jobs / automations: No contract impact.

Risk Register Notes: Deep. Mutates `users.passwordHash` directly via bcrypt compare-then-rehash. Revokes ALL of the caller's active refresh tokens on success (same theft-cascade precedent as `auth.service.ts#refresh()`'s reuse-detection branch), forcing re-login on every other device/tab — this is intentional defense-in-depth, not a bug. Required checks: current-password verification via `bcrypt.compare` before any mutation; new-password validated with the same `@MinLength(8)` rule as `RegisterDto`; revocation scoped to exactly the caller's `userId` (never another user's tokens); server clears the caller's own `mnemra_rt` cookie so the current tab doesn't retry a now-dead refresh token.

## Notes
- `RT_COOKIE` constant, `setRtCookie`/cookie-clearing pattern, and `RegisterDto`'s `@MinLength(8)` rule already exist in `auth.controller.ts`/`auth.service.ts`/`register.dto.ts` — reuse exactly, do not invent new conventions.
- `BCRYPT_ROUNDS = 12` constant already exists in `auth.service.ts` — reuse it for the re-hash, do not hardcode a new value.
- Revocation query pattern to mirror: `auth.service.ts#refresh()`'s theft-cascade branch — `db.update(refreshTokens).set({revokedAt: now}).where(and(eq(refreshTokens.userId, ...), isNull(refreshTokens.revokedAt)))`. Wrap the passwordHash update + revocation together (transaction not strictly required since both are idempotent single-table writes with no cross-table invariant, but check `auth.service.ts`'s existing style before deciding — `refresh()`'s rotation uses a transaction for a similar two-write sequence, so match that precedent unless there's a clear reason not to).
- Controller must clear the `mnemra_rt` cookie server-side (`res.clearCookie(RT_COOKIE, {path:'/'})`) in addition to the DB-side revocation, mirroring `logout()`'s pattern, so the current tab doesn't retry a dead token before the frontend's own redirect kicks in.
- Frontend flow on success: toast → `await logout()` (clears any remaining client-side state) → `router.push('/login')`. On 401 (wrong current password): inline field error only, do NOT log out or redirect — this is a validation failure, not a session-invalidation event.
- `CurrentUserContext` is `{userId, email}` (see `current-user.decorator.ts` / `jwt.strategy.ts`) — use `user.userId`, not `user.sub`.
