## Vision

### 10x Check

The validated ask is narrow (transcript → ticket for ELID), and that narrowness is correct right now — the 10x version isn't a bigger feature, it's the same feature proven repeatable: an extraction chain generic enough that "conversation → structured record" works for Slack threads and meeting notes too, once ELID's pilot validates the pattern with call transcripts first. Scope stays narrow; the architecture stays loosely coupled enough not to foreclose that path.

### Platonic Ideal (12 months)

Mnemra has two proven product surfaces sharing one auth/workspace substrate: chat-RAG over ingested docs, and the ticket-copilot extraction pattern generalized beyond call transcripts. Neither was built ahead of validated demand — each shipped narrow, proved itself with a real customer, then earned its next increment.

## Scope Decisions

| # | Proposal | Effort | Decision | Reasoning |
|---|----------|--------|----------|-----------|
| 1 | Platform framing (shared workspace/billing across chat-RAG + ticket-copilot from day one) | L | REJECTED (kept Approach A) | Only ELID's narrow ask is validated; platform framing bets on an unvalidated second wedge and adds coupling cost now for a future that may not arrive. Substrate reuse (auth/DB) already keeps the door open without paying that cost. |
| 2 | Per-field "regenerate" (re-extract one field instead of the whole draft) | S | ACCEPTED | Cheap, same extraction chain, meaningfully reduces re-review friction. Regenerating a field also re-runs that field's confidence score (see #4) — the two are linked, not independent. |
| 3 | "Copy as Linear markdown" formatting (headers, checklists for repro steps) | S | ACCEPTED | Trivial formatting work, directly improves the manual-copy v1 default's usability. Mapping: title → H2 header, issue summary → plain paragraph, repro steps → checklist, severity/product area → bold label line, hypothesized root cause → italicized "guess" line, next action → plain paragraph. |
| 4 | Extraction confidence summary badge ("4/6 fields high confidence") | S/M (dependency, see below) | ACCEPTED | The extraction prompt/schema currently scores confidence for one field (`hypothesizedRootCause`) only. Extending this to all 7 fields is prompt/schema work, not just aggregation of existing data — get a real estimate before implementation treats this as trivial. The existing single-field confidence display is reused/extended, not rebuilt. |
| 5 | Audit trail (reviewedBy, reviewedAt columns using existing JWT auth) | S | ACCEPTED | Trivial with auth already in place; useful even for a single-reviewer pilot as a paper trail. Semantics: last-touch, not a history log — `reviewedBy`/`reviewedAt` update on every save (including per-field regenerate from #2), overwriting the prior value. A full audit history is out of scope (see Deferred). |
| 6 | Duplicate-transcript detection via content hash before LLM call | S | ACCEPTED | Saves LLM cost and avoids duplicate ticket drafts from accidental re-paste. Exact-match only (whitespace-trimmed, line-endings-normalized hash) — no fuzzy/near-duplicate detection in v1, that's a known gap, not an oversight. UX: show a banner ("This transcript was already submitted — showing the existing ticket") and route to the existing ticket rather than silently redirecting or blocking. Requires a new indexed `transcriptHash` column on the `tickets` table (schema change, alongside #5's columns — not an existing column being reused). Fallback: if the matched ticket was since deleted/archived, fall through to normal extraction rather than erroring. |
| 7 | Route extraction through the existing Bull/Redis queue (apps/api already uses it for document ingest) instead of a synchronous request/response call | M | ACCEPTED | Synchronous OpenAI calls are a scaling bottleneck beyond single-pilot volume. Reuses existing job infra rather than inventing new resilience patterns. Changes the review screen's loading state from a synchronous wait to an async job-status poll/push (see Design Review's loading state — needs a revisit for job-based UX, not just a spinner). |

## Accepted Scope (added to this plan)

- Per-field regenerate action on the review/edit screen (also re-scores that field's confidence)
- Linear-markdown-formatted clipboard copy per the field→element mapping above
- Per-field confidence scoring extended to all 7 ticket fields (S/M effort — real prompt/schema work, not just aggregation; existing root-cause confidence display is extended, not rebuilt), aggregated into a top-of-draft confidence badge
- `reviewedBy` / `reviewedAt` columns on the `tickets` table (last-touch semantics, updated on every save), populated from the authenticated user
- Transcript content-hash check (exact-match, normalized) before the extraction LLM call; on match, show a banner and route to the existing ticket. New indexed `transcriptHash` column required. Falls through to normal extraction if the matched ticket was deleted/archived.
- Extraction routed through the existing Bull/Redis queue instead of a synchronous call; review screen's loading state becomes an async job-status poll/push rather than a synchronous wait

## Security & Reliability (added during /plan-ceo-review 11-section review)

- IDOR fix: every ticket query scoped by `workspaceId` from JWT, not just ticket ID
- Prompt-injection hardening added to the extraction prompt
- 4 error-handling gaps closed (JSONParseError, ModelRefusalError, ConnectionPoolExhausted, duplicate-hash DbQueryError) — see full design doc's Error & Rescue Registry
- Key metric: feedback usefulness rate (not just technical job-success rate)
- No feature flag — ships live for all workspaces (pre-launch product)
