## 2026-07-10 — Optra A3 (vendor catalog matching): pdfjs-dist real render only works under Vitest/plain Node, not Jest
**Predicted:** no live prediction solicited — standing session directive to skip live predict-verify for speed, log real discoveries instead. Going in: `catalog.e2e-spec.ts` would upload a real `pdf-lib`-generated PDF (same pattern as A2's `pdf-render-integration.spec.ts`) and the real `renderPdfToImages` chain would just work end-to-end through the Jest e2e suite, since it already works under Vitest.
**Actual:** It didn't. Real (non-mocked) `renderPdfToImages` threw `Cannot use 'import.meta' outside a module` the moment Jest's module loader tried to load `pdfjs-dist`'s legacy `.mjs` build via its dynamic `import()`. Verified this is Jest-specific, not a real bug, by testing the exact same compiled `packages/ai/dist` output directly under plain Node (`node -e`, real `pdf-lib` PDF, real render) — it worked immediately. Root cause: TypeScript compiles `await import(...)` to a wrapped `require()` when targeting CommonJS (`packages/ai/tsconfig.json`), and plain Node has native `require(esm)` support that lets that `require()` transparently load a real ESM `.mjs` file — but Jest's own module system doesn't go through Node's native loader at all, so it can't. This had never surfaced before because A2's own e2e (`procurement.e2e-spec.ts`) fully overrides `ProcurementExtractionService`, so it never actually reaches the real rasterizer either — A3's e2e is the first place in this repo any test exercises this real call chain through Jest specifically.
**Why different:** Fixed in the test file only, not the source: `catalog.e2e-spec.ts`'s Jest-run happy path uses the CSV/`photo_url` upload branch instead of PDF (with `CatalogImageService` also overridden to avoid a real network fetch) — proves the exact same full pipeline (upload→queue→processor→items→search→match→dismiss) without touching the Jest-incompatible code path. Considered and rejected two other fixes: adding Jest ESM config (`--experimental-vm-modules` + `extensionsToTreatAsEsm`) has a large, repo-wide blast radius across every other `apps/api` spec for a benefit only this one test needs; changing `packages/ai`'s module target away from CommonJS would ripple through every other consumer of `@repo/ai`'s compiled dist. The PDF page-render branch itself stays fully proven at the only two layers that actually can exercise it for real (Vitest's `pdf-render-integration.spec.ts`, Jest's `catalog-parse.processor.spec.ts` with the render step mocked) — this is a test-vehicle substitution, not a coverage gap.

## 2026-07-10 — Optra A2 (PDF extraction, vision reversal: mupdf → pdfjs-dist/@napi-rs/canvas)
**Predicted:** no live prediction solicited (standing session directive) — user first said "accept AGPL for mupdf anyway... it's just a prep project for a job application," then asked "and we can't do that without mupdf?", prompting the actual resolution: the concept ("screenshot pages, send to OpenAI") never needed mupdf specifically, just *some* rasterizer.
**Actual:** Swapped to `pdfjs-dist@6.1.200` (Apache-2.0, Mozilla's own PDF.js — already indirectly in this repo's tree via `pdf-parse`) + `@napi-rs/canvas@1.0.2` (MIT, prebuilt N-API binaries, verified license/binary-distribution claims via web research before installing, not assumed). `mupdf` fully removed; zero AGPL dependency in the shipped result — a better outcome than "accept AGPL for now," not just an equivalent one. Two real API surprises, both caught by verifying against the actually-installed package rather than trusting general library knowledge: (1) `pdfjs-dist`'s `getDocument()` throws if handed a Node `Buffer` directly — needs an explicit `new Uint8Array(data)`, since it does a real `instanceof Buffer` check despite `Buffer` extending `Uint8Array`; (2) this pdfjs-dist version's `render()` `RenderParameters` type requires `canvas: null` explicitly alongside `canvasContext` (the API shifted — `canvas` is now primary, `canvasContext`-only is the documented backwards-compat path and requires the null). Both were caught by the type-checker and a plain `bun run` smoke script *before* trusting any test framework — deliberately, since `vitest`/`esbuild` had just spent 20+ minutes hung on the *previous* library (`mupdf`'s top-level `await` WASM init blocking Vite's dependency scanner, verified directly in `node_modules/mupdf/dist/mupdf.js`) and left multiple unkillable zombie `esbuild --service` processes in `UE` (uninterruptible sleep) state that `pkill -9` couldn't touch. Also discovered mid-session: this conversation's own scratchpad path and background-task/Fact-Forcing-Gate counters silently reset at some point (compaction boundary) without an explicit notice — a stale scratch-file path from before the reset caused a confusing "module not found" a few turns later.
**Why different:** Three lessons, not one diff. (1) A user pushing back with "can't we do X without Y" is worth taking at face value as a real question, not just reassurance-seeking — the honest answer here was "yes, obviously," and re-examining the *actual requirement* (screenshot pages → vision model) instead of the *specific tool* I'd fixated on (`mupdf`) produced a strictly better outcome. (2) When a test runner itself is misbehaving (not the code under test), the right move is to stop feeding it the same command and switch to the most primitive verification channel available (`bun run` a plain script) rather than escalating retries with longer timeouts — this is what actually produced fast, real, trustworthy feedback today, twice. (3) After a long session, don't assume paths/IDs mentioned early in the conversation are still valid — a compaction event can silently rotate session-scoped state (scratchpad dir, gate counters); re-derive from the *current* system prompt rather than conversation history when a path stops working.

## 2026-07-10 — Optra A2 (PDF extraction, Step B/C: rasterizer + LLM chain)
**Predicted:** no live prediction solicited — user directed bypassing live predict-verify for this session ("needed to be done fast"), asking only that this log stay current as a handbook. Documenting real discoveries instead.
**Actual:** Step B (scanned/image PDF support via `mupdf` WASM rasterizer + gpt-4o vision) never got built — `mupdf`'s own `LICENSE` file, read directly after `bun add mupdf`, is AGPL-3.0-or-later. For a closed-source multi-tenant SaaS, AGPL's network-use clause is real legal exposure, not a technical footnote, so implementation stopped and the finding went to the user with options (buy a commercial license, find a permissive-license rasterizer, ship text-only, or knowingly accept AGPL). Decision: **ship text-only A2**, `mupdf` reverted (`bun remove mupdf`), scanned/image PDFs now fail loudly with an explicit "not supported yet" error instead of silently producing nothing. Step C (`packages/ai/src/chains/procurement-extraction.ts`) mirrors `ticket-extraction.ts` almost verbatim (prompt→JSON→hand-validate→typed errors, temperature 0, 2-attempt timeout retry) — the actual novelty was domain (procurement's first-ever OpenAI call), not code shape. TDD caught a real bug before it shipped: the first draft only checked "is quantity a non-empty string," so an LLM hallucination like `quantity:"ten"` would have passed validation and then failed the *entire* bulk insert at the Postgres `numeric` column boundary — one bad row from the model would have silently failed a whole otherwise-good document. Fixed with a `nullableNumericString` check (`Number.isFinite(Number(trimmed))`) so malformed numeric fields null out individually instead of poisoning the batch.
**Why different:** Two tradeoffs, not one diff. (1) License-checking a new dependency's actual `LICENSE` file (not just assuming "WASM = safe") should happen *before* recommending it in a plan, not after `bun add` — the plan's own dependency-risk analysis was purely technical (ABI/native-binary safety) and missed a real legal dimension entirely; this is now a gap worth checking earlier next time a new package is proposed. (2) Field-level numeric validation at the LLM-output boundary isn't "unnecessary validation for a scenario that can't happen" (which CLAUDE.md warns against) — an LLM returning a non-numeric string in a numeric-shaped field is a well-known, common extraction failure mode, and the blast radius (one bad row failing a whole bulk insert) is concrete and immediate, not speculative — validating external-API output at the boundary is exactly the case CLAUDE.md's own guidance carves out as warranted.

## 2026-07-08 — V2 slice S1+F1 (structured CSV querying / DuckDB)
**Predicted:** No user prediction was solicited for this slice specifically — implementation proceeded under a broad "continue autonomously, ensure the plan is implemented correctly" instruction rather than a per-slice Learning Contract prediction. Documenting real discoveries instead.
**Actual:** The approved plan's hardening design (physical isolation + `enable_external_access=false` + keyword filtering) was implemented as designed, but empirically verified with a throwaway spike BEFORE writing the real service, not assumed correct from the plan text alone.
**Why different:** Verification surfaced something better than the plan anticipated — DuckDB doesn't just block filesystem access after lockdown, it refuses to let `enable_external_access` be re-enabled for the connection's lifetime ("Cannot change enable_external_access setting while database is running"), a genuine engine-level one-way lock. That doesn't remove the need for the keyword-based DDL/DML filter (CREATE/DROP/INSERT/etc. still work against the already-loaded in-memory table without needing filesystem access), but it meant one layer of the defense-in-depth design was stronger than assumed. Separately, two real bugs only surfaced under test: (1) DuckDB's Node driver returns aggregate results (`SUM`, etc.) as JS `bigint`, which crashes `JSON.stringify` — sanitized to `Number` before results leave `DuckDbQueryService`; (2) drizzle-kit generates an invalid quoted `"vector(1536)"` type declaration for this pgvector customType in fresh migrations (same issue hit in S3's migration) — now a documented, repeatable fix rather than a surprise each time.

## 2026-07-08 — V2 slice S3 (chat_query_metrics telemetry)
**Predicted:** "New table storing chat query metrics to be used later for related features such as auto FAQ generation."
**Actual:** New additive `chat_query_metrics` table (correct), but the real work was wiring a fire-and-forget-shaped write hook into the three-branch hot chat path without changing existing behavior: exporting `classifyQuery` from the `@repo/ai` barrel (it wasn't public), making `persistAssistant()` return the new message id, skipping embedding on exact-cache hits to avoid extra OpenAI cost on the cheapest path, and hand-fixing a drizzle-kit-generated `"vector(1536)"` (quoted, invalid) to the unquoted form the two existing pgvector migrations use.

## 2026-07-08 — Post-login/OTP redirect to /chat inconsistent
**Predicted:** User's own bug report doubled as the prediction: "once we get a successful API response it should redirect the user to the protected /chat page."
**Actual:** The API response and redirect call were already correct and always fired — `login()`/`verifyOtp()` resolve, `router.push('/chat')` runs every time. ## 2026-07-08 — Silent mid-session logout on expired mnemra_at
**Predicted:** Retry-on-401 in `apiFetch` would need to call the existing-but-unused `refreshAccessToken()` from `auth.ts`, since it already wraps `POST /api/auth/refresh` and re-using it avoids duplicating that call.
**Actual:** Implemented a small standalone `fetch('/api/auth/refresh')` directly inside `client.ts` instead of importing `refreshAccessToken()` from `auth.ts`.
**Why different:** `auth.ts` already imports `apiFetch` from `client.ts` — if `client.ts` also imported `refreshAccessToken` from `auth.ts`, that's a circular module dependency between the two files. It would likely still work at runtime here (neither usage happens at module-init time), but it's a fragile pattern to introduce for saving four lines of duplicated code, and a circular dependency between a shared HTTP client and a single API-domain module is the kind of thing that gets harder to reason about as more retry/interceptor logic gets added later. Kept `client.ts` self-contained instead; `refreshAccessToken()` stays in `auth.ts`, unused, as a public export for any future explicit "manually refresh" call site. Also worth noting: the fix deliberately does NOT try to distinguish "refresh succeeded but retry still failed" from "refresh failed outright" in a special way — both just throw and let the existing per-page `isUnauthorized()` handlers redirect to `/login`, since a genuinely revoked session should still end in a redirect, not a silent retry loop.
