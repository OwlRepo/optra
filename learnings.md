## 2026-07-08 — V2 slice F7a (coverage dashboard, built ahead of F6)
**Predicted:** no live user prediction solicited — user said "proceed on the next one" via `/ecc:prp-implement`; I flagged F6's two real blockers (Resend domain unverified, zero accumulated telemetry) and built F7a instead, documenting the substitution rather than asking a fresh question.
**Actual:** Panels 1+2 (summary, low-score list) are trivial live rollups. Panel 3 (topic gaps) reuses `FaqClusterService.cluster()` completely unchanged for a second, unrelated purpose (clustering chat-metric ids by question-embedding similarity instead of ticket ids) — the function only ever touches `.embedding`, so the "ticketId" field name is just a slot, not a real coupling to tickets. Storage is Redis, not a new table, matching the plan's own "no migration" classification for this slice — this repo's first read-model that intentionally lives outside Postgres.
**Why different:** Reusing `FaqClusterService` instead of writing a near-identical clustering function again was a deliberate DRY call given how small and pure the function is (no DB/LLM access inside it), rather than "premature abstraction" — the abstraction already existed and was proven by 4/4 passing tests before this slice touched it. Choosing Redis over a new `topic_gaps` table was driven by the plan's explicit "no migration" line for F7a: a cache that occasionally goes empty (Redis flush/eviction) and refills on the next weekly run is a materially different risk profile than a table would have been, and that tradeoff is stated in the risk register rather than left implicit.

## 2026-07-08 — V2 slice F5 (cross-file comparison queries)
**Predicted:** no live user prediction solicited — user said "go for f5" after the standing "stick to plan, avoid back-and-forth" instruction; documenting real discoveries instead.
**Actual:** Deviated from the plan's literal "multi-select disambiguation UX" — a checkbox-style multi-select would have needed a new frontend interaction pattern this repo doesn't have anywhere else. Instead, comparison intent is detected from the question text itself (`classifyComparisonIntent`), and when 2+ dataset candidates score above a looser threshold than the single-dataset confident bar, they're all loaded and joined automatically — no disambiguation step needed at all for the common case ("compare sales vs refunds" already names what to compare). `DuckDbQueryService.runReadOnlyQuery()` was refactored to delegate to a new N-table method rather than duplicating the hardening logic, keeping the public single-table signature untouched for F1/F2's existing callers.
**Why different:** The plan was written before any of S1's actual code existed, when it wasn't yet obvious that natural-language comparison questions already carry enough signal to skip UI-level disambiguation entirely — building the UX anyway would have been speculative complexity for a case the heuristic already resolves. Chose a looser comparison-specific score threshold (0.35 vs. 0.5) as its own tunable constant rather than reusing `CONFIDENT_MIN_SCORE`, since a two-topic question's cosine similarity against either single dataset is structurally lower than a one-topic question's — conflating the two thresholds would have made single-dataset matching either too strict or comparison-matching too loose.

## 2026-07-08 — V2 slice F4 (auto-FAQ from ticket clusters)
**Predicted:** no live user prediction solicited — user directed continuation ("proceed on the next if F3 is all done") after confirming the standing "stick to plan, avoid back-and-forth" instruction; documenting real discoveries instead.
**Actual:** The plan's own "after F3 soak" sequencing note turned out to be non-blocking once `TicketDocCoverageService` was actually built: `findGaps()` (F3, doc-centric) and the new `findUncoveredTickets()` (F4, ticket-centric) are two reads of the SAME query shape, so F4 could start immediately rather than waiting on calendar time — the plan was written before implementation, when it wasn't yet clear the shared primitive would double as both views for free. Clustering is a plain greedy single-pass algorithm (cosine vs. a fixed seed embedding, not a recomputed centroid) — deliberately simple over a proper clustering library, since the input size per workspace per week is small and determinism matters more than clustering quality here. FAQ approval reuses `scrape.processor.ts`'s document-creation pattern exactly (storage write → `documents` insert → `IngestService.queueDocument()`), which meant zero new ingest-pipeline code was needed — the "materializes as a normal documents row" language in the plan was literally already a solved problem elsewhere in the codebase.
**Why different:** Chose to build the coverage-primitive extension as a second exported method (`findUncoveredTickets`) rather than modifying `findGaps()`'s existing query — F3 was already shipped/tested/committed, and CLAUDE.md's Backward-Compat discipline favors additive extension of a shared file over touching working code, even when the two methods share ~80% of their SQL. Chose PII mitigation via already-extracted ticket fields (not a redaction pass over raw transcripts) because those fields already exist, are already reviewer-approved, and are a structurally smaller surface than free text — cheaper and safer than building new redaction logic.

## 2026-07-08 — Docker ARM64 fix + V2 slice S2+F3 (scheduler + freshness detector)
**Predicted:** no live user prediction solicited — user delegated the Docker fix choice ("you decide... as long as it will not break anything and the deployment docker in vps still a success") and directed autonomous continuation into F3 ("stick to the original plan to avoid back and forth").
**Actual:** Docker fix: root-caused that the earlier "ARM64 gap" only ever affected local Apple Silicon Docker builds, never production — `.github/workflows/deploy.yml` builds directly on the Hetzner VPS over SSH (not on the GitHub-hosted runner), and `DEPLOYMENT.md:46` confirms that VPS is a CX21 (native x86_64). Fixed by pinning `platform: linux/amd64` on the `api` service in both compose files — a no-op in production, forces correct emulation locally. F3: resolved the plan's one open point (embedding-distance vs LLM-compare for freshness) in favor of embedding-distance, reusing the already-computed ticket/document chunk embeddings via a `LATERAL` cosine join — zero new LLM cost. Built the S2 scheduler substrate as this repo's first Bull repeatable job (`FreshnessTickProcessor`, weekly cron, `background_runs` bookkeeping) fanning out one per-workspace `freshness-check-queue` job, and F3's `document_review_flags` table (N-per-doc with ticket/score provenance, audited dismiss).
**Why different:** The Docker "fix" turned out to be a documentation/verification correction more than a code fix — the actual production risk was already zero, but the compose files previously left platform ambiguous (relying on host default), which is a latent risk on any future non-x86_64 host; pinning explicitly removes that ambiguity rather than just patching the symptom. For F3, choosing embedding-distance over LLM-compare wasn't just cost-driven: the `NOT EXISTS`-guarded `LATERAL` join can run as a single SQL statement per workspace with no round-trips, matching the batch's established "cheap-first, LLM only for genuinely ambiguous cases" pattern (classify.ts's two heuristics) rather than introducing a new per-ticket LLM call class that would also need token-budget accounting.

## 2026-07-08 — F1 frontend/XLSX finish + Docker root-cause chase
**Predicted:** no prediction solicited — user directed "do both now, make sure it's done first."
**Actual:** Frontend + XLSX landed cleanly. The Docker verification chased through FOUR distinct real bugs before revealing a genuine architectural blocker: (1) Alpine musl can't load DuckDB's glibc binary, (2) fixed base image but then discovered `docker compose build` was silently building into a detached buildx builder instance and never loading the result into the local image store (my own exit-code check was also wrong — `cmd | tail` reports tail's exit code, not cmd's, which hid a real build failure for two attempts), (3) once building correctly, Debian's `--no-install-recommends` skipped `ca-certificates`, breaking all HTTPS installs, (4) once THAT was fixed, the real blocker: no Linux ARM64 prebuilt binary exists for `duckdb@1.4.4`, and there's no C++ toolchain to compile from source.
**Why different:** Each layer looked like "the" bug until verified end-to-end — a lesson in not declaring victory on the first plausible-looking fix. Left the two genuine improvements (Debian base, ca-certificates) in place since they're correct regardless; did not attempt a toolchain install or platform-forcing fix without the user's input, since compiling DuckdB from source or changing target architecture materially changes build time/deploy shape and deserves an explicit decision.

## 2026-07-08 — V2 slice S1+F1 (structured CSV querying / DuckDB)
**Predicted:** No user prediction was solicited for this slice specifically — implementation proceeded under a broad "continue autonomously, ensure the plan is implemented correctly" instruction rather than a per-slice Learning Contract prediction. Documenting real discoveries instead.
**Actual:** The approved plan's hardening design (physical isolation + `enable_external_access=false` + keyword filtering) was implemented as designed, but empirically verified with a throwaway spike BEFORE writing the real service, not assumed correct from the plan text alone.
**Why different:** Verification surfaced something better than the plan anticipated — DuckDB doesn't just block filesystem access after lockdown, it refuses to let `enable_external_access` be re-enabled for the connection's lifetime ("Cannot change enable_external_access setting while database is running"), a genuine engine-level one-way lock. That doesn't remove the need for the keyword-based DDL/DML filter (CREATE/DROP/INSERT/etc. still work against the already-loaded in-memory table without needing filesystem access), but it meant one layer of the defense-in-depth design was stronger than assumed. Separately, two real bugs only surfaced under test: (1) DuckDB's Node driver returns aggregate results (`SUM`, etc.) as JS `bigint`, which crashes `JSON.stringify` — sanitized to `Number` before results leave `DuckDbQueryService`; (2) drizzle-kit generates an invalid quoted `"vector(1536)"` type declaration for this pgvector customType in fresh migrations (same issue hit in S3's migration) — now a documented, repeatable fix rather than a surprise each time.

## 2026-07-08 — V2 slice S3 (chat_query_metrics telemetry)
**Predicted:** "New table storing chat query metrics to be used later for related features such as auto FAQ generation."
**Actual:** New additive `chat_query_metrics` table (correct), but the real work was wiring a fire-and-forget-shaped write hook into the three-branch hot chat path without changing existing behavior: exporting `classifyQuery` from the `@repo/ai` barrel (it wasn't public), making `persistAssistant()` return the new message id, skipping embedding on exact-cache hits to avoid extra OpenAI cost on the cheapest path, and hand-fixing a drizzle-kit-generated `"vector(1536)"` (quoted, invalid) to the unquoted form the two existing pgvector migrations use.
**Why different:** A genuinely fire-and-forget (`.catch()`, not awaited) write created a real race — `onComplete` could resolve before the insert landed, invisible in production (nobody observes the timing) but a flaky/failing assertion in tests, and more fragile in general for a per-request Node handler with no background-job supervisor. Switched to `await write().catch(handler)`: still isolates failures from the response (never rejects), but now deterministic. Also decided nullable embeddings over always-embedding, since exact-cache hits are the least interesting rows for the topic-gap clustering this table exists to feed (F7a) — not worth an extra embedding call on every cache hit just for telemetry completeness.
